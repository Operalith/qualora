# Qualora v0.21.0-alpha

`v0.21.0-alpha` adds policy-gated AI Browser Control.

## Highlights

- Start AI Browser Control runs for projects with a `frontend_url`.
- Ask an OpenAI-compatible provider for one typed JSON browser action at a time.
- Validate every suggestion with Qualora's deterministic safe-browser policy before Playwright executes anything.
- Record sanitized observations, AI suggestions, policy decisions, execution outcomes, screenshot evidence, findings, trace data, JSON reports, and self-contained HTML reports.
- Use the web UI to start runs and inspect policy timelines.
- Use `fake-llm` smoke fixtures for a deterministic safe path and a policy-blocked unsafe suggestion.

## Safety Model

AI Browser Control does not give the model direct Playwright control. The model can only suggest a supported action, and Qualora executes only approved safe actions.

Supported action types:

- `goto`
- `click_link`
- `click_safe_navigation`
- `assert_text_visible`
- `assert_url_contains`
- `assert_title_contains`
- `capture_screenshot`
- `collect_browser_signals`
- `stop`

Qualora does not send credentials, cookies, browser storage, auth headers, tokens, screenshots, full HTML, request bodies, or response bodies to AI. Form submission, arbitrary unsafe button clicks, external crawling by default, active scanning, payload execution, fuzzing, and destructive actions remain blocked.

## Known Limitations

- AI Browser Control is alpha and conservative.
- Only OpenAI-compatible chat completion providers are supported.
- The action schema is intentionally small.
- No arbitrary selectors, form submission, payload attacks, fuzzing, destructive testing, active scanning, or autonomous/direct AI browser control.
- Safe QA integration is standalone in this release; AI Browser Control has its own run/report endpoints.
- Reports and comparisons do not replace human review.
