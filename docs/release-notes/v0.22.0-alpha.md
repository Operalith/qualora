# Qualora v0.22.0-alpha

Qualora v0.22.0-alpha adds Safe Form Testing: deterministic form understanding plus bounded execution for safe same-origin GET forms such as search, filter, sort, and navigation/query forms.

This release keeps the safety boundary conservative. Qualora understands more forms, but it still does not submit arbitrary forms, mutating forms, password forms, payment forms, uploads, account/admin mutation forms, destructive forms, fuzzed inputs, or payload attacks.

## Highlights

- Project-scoped Safe Form Testing runs.
- Form classification for search, filter, sort, newsletter, contact, login, password, payment, profile update, upload, admin mutation, destructive, and unknown forms.
- Safe execution only for same-origin GET forms classified as safe.
- Bounded deterministic test values, such as `demo`, safe select defaults, small numbers, and stable dates.
- Skipped-form records with explicit reasons for unsafe, unsupported, external, sensitive, mutating, or destructive forms.
- `form_observations`, `form_submission`, and screenshot evidence metadata.
- JSON and self-contained HTML Safe Form Testing reports.
- Report intelligence support for Safe Form Testing findings.
- Web UI project controls, run lists, report pages, and reports landing integration.
- AI Browser Control `submit_safe_get_form` support, gated by deterministic policy.
- Demo web safe/unsafe form fixtures and fake LLM safe/unsafe form suggestions.
- Smoke coverage for form testing, policy-approved form suggestions, policy-blocked unsafe form suggestions, reports, UI text, and redaction.

## API

New endpoints:

- `POST /api/v1/projects/{project_id}/form-test-runs`
- `GET /api/v1/projects/{project_id}/form-test-runs`
- `GET /api/v1/form-test-runs/{form_test_run_id}`
- `GET /api/v1/form-test-runs/{form_test_run_id}/report`
- `GET /api/v1/form-test-runs/{form_test_run_id}/report.html`

Example request:

```bash
curl -s -X POST "http://localhost:8080/api/v1/projects/${PROJECT_ID}/form-test-runs" \
  -b "$COOKIE_JAR" -c "$COOKIE_JAR" \
  -H "X-Qualora-CSRF: ${CSRF}" \
  -H "Content-Type: application/json" \
  -d '{
    "use_latest_discovery": true,
    "target_url": "http://demo-web:8080/",
    "max_forms": 12,
    "max_tests_per_form": 1,
    "safe_get_only": true
  }'
```

## Safety

Safe Form Testing executes only forms that satisfy all of these checks:

- Method is `GET`.
- Action URL is same-origin and allowed by project `allowed_hosts`.
- Classification is safe search, filter, sort, or navigation-like behavior.
- Fields are non-sensitive and can be filled with bounded deterministic values.
- No password fields, file fields, hidden sensitive fields, destructive labels, destructive action paths, or sensitive parameter names are present.

Skipped by default:

- `POST`, `PUT`, `PATCH`, `DELETE`, and other mutating methods.
- Login, password reset, payment, checkout, transfer, refund, delete, cancel, deactivate, upload, account/profile/admin mutation, destructive, unknown, and unsupported forms.
- External form actions.
- Sensitive field names such as password, token, secret, api_key, card, cvv, account, auth, and session.

Qualora does not store raw submitted values, request bodies, response bodies, cookies, browser storage, auth headers, tokens, credentials, provider secrets, or full HTML in Safe Form Testing reports or evidence.

AI Browser Control may suggest `submit_safe_get_form`, but Playwright executes it only after deterministic policy approval. AI never receives credentials, cookies, storage, auth headers, tokens, screenshots, full HTML, request bodies, response bodies, or raw form values.

## Smoke Coverage

`make smoke` now verifies:

- Safe Form Testing run creation and completion.
- Safe GET search form execution against `demo-web`.
- Unsafe, mutating, dangerous, external, and unsupported forms skipped with reasons.
- Safe Form Testing JSON and HTML reports.
- Screenshot evidence download.
- Report intelligence fields.
- No demo passwords, API tokens, raw form values, cookies, storage, auth headers, request bodies, or response bodies in reports.
- AI Browser Control safe form suggestion approval.
- AI Browser Control unsafe form suggestion block.
- Existing browser, authenticated browser, API, authenticated API, AI analysis, AI planning, safe plan execution, authorization, discovery, quality, Safe Explorer, Safe QA, baseline/gate, CI, issue export, and guided setup flows.

## Known Limitations

- Safe Form Testing is alpha and intentionally narrow.
- Only safe same-origin GET forms are executed by default.
- Safe QA integration is standalone for this release; form reports are separate from Safe QA Runs.
- No arbitrary form submission.
- No POST/mutating form submission.
- No multi-step form workflows.
- No MFA or autonomous login automation.
- No fuzzing, payload attacks, active security scanning, destructive testing, or full form coverage.
- AI Browser Control remains policy-gated and does not provide direct/free-form AI browser control.
